# Milestone planning + release (how this repo works)

Work is shipped as a sequence of **milestones**, each a single reviewed PR. Never open
a second PR while one is still open — the milestones touch shared core files
(`detector.py`, `harness.py`, `corpus_builder.py`, `tests/test_pcs.py`) and would collide.
This cadence has shipped every change here (PRs #6–#11) with all CI green.

## Before — plan by milestone

- Break the work into milestones; make each one **ultra clear** about what it changes and
  why it matters. Present the plan and **get approval before writing code**. For a
  multi-milestone roadmap, execute one milestone at a time and check in between.
- State the **open-PR check** up front: there must be **0 open PRs** before a milestone
  starts (one PR at a time).

## During — implement + review (the review gate is not optional)

1. Branch off fresh `main` (`git checkout main && git pull && git checkout -b <milestone>`).
2. Implement the milestone.
3. Run a **high-effort code review** — `/code-review` (the multi-agent Workflow-backed one).
   It reliably finds real bugs here (it has caught a 95-second CPU-DoS, an unwinnable regex
   denylist, `re.IGNORECASE` false positives, a silently-ignored `--framework=` flag).
   **Fix every confirmed finding before merging.** For risky changes, also spawn a focused
   adversarial probe agent to try to break the fix.
4. Green locally, exactly as CI does:
   ```bash
   uv run pytest tests/test_pcs.py -q --noconftest   # local needs --noconftest
   uv run black --check .
   ```

## After — release (industry-standard, cautious)

5. Push the branch; open **one** PR with `gh pr create` (descriptive title + body).
6. Wait for **all four** CI checks green: `test`, `General Code Linting`, `GitGuardian
   Security Checks`, `Markdown Linting`. (See [ci-triage](../ci-triage/SKILL.md) if any go red.)
7. `gh pr merge <#> --merge --delete-branch`; then `git checkout main && git pull`; delete
   the local branch; verify `main` is green. Report the merged PR URL.

## Hard guardrails (do not cross)

- **Never commit generated corpus data.** `pretense_compliance_standards/corpus/*.json`
  (except the small committed base files already tracked) and everything under
  `frameworks/*/{corpus,database,codebase,data,logs}/` are **git-ignored build artifacts**
  regenerated by `python -m pretense_compliance_standards.corpus_builder`. They hold
  synthetic `sk_test_…` / PEM values that trip **GitHub push protection**. Commit only
  source + the `frameworks/**/README.md` files. If a `git add -A` stages regenerated
  corpus, restore it before committing.
- **Everything is synthetic and provably fake** (900-range SSNs, `@example.com`, `sk_test_`
  keys, RFC-5737 IPs, `00`-check IBANs). Every generated file carries the
  `SYNTHETIC — FAKE COMPLIANCE TEST DATA, NOT REAL` banner.
- **No compliance-framework names inside a scanned `text` payload** — only in metadata,
  reports, table/variable/file names, and docs.
- Preserve Jeroen Bouma's upstream MIT LICENSE + attribution; do not gut `financedatabase/`,
  `database/`, `compression/`.

## Push protection

Don't allowlist a secret to get a push through. Keep synthetic secret-shaped values **out of
the committed diff** (generate + git-ignore them) — that is why the guardrail above exists.
